5 Summary of Checklists >>>work in progress<<<





Which button is active by default?

  • In normal dialogs it should be the 'Next' button.

  • In message boxes normally the 'left' button (= go on).

  • In message boxes where user can delete something or can do a critical action the button that will abort the action (normally the 'right' button 'Cancel')

Single check box

  • Does the default (checked or not checked) make sense?

Radio buttons (2 to more): Which radio button is active by default?

  • It should be the first in the column

  • or if not possible the selection user will use in most cases

Cycle through the dialog with the tab-key (Shift-tab = move backwards)

  • Is the order of the tabs consistent to the expectation of the user? Normally from top to down, or is there a 'strange' move?

  • Do you see where the focus is (e.g. button with special frame, dotted line round check box, ....)?

Is there a text cut caused by the frame of the field?

  • Check the text on the buttons and in the gray inserted text fields

Check the function of short cut keys for all

  • buttons

  • radio buttons with text

  • check boxes

  • combo boxes

  • list entry fields

  • editable text fields

  • ect.

Check if there are double short cut keys!

Check the function of arrow-keys

  • Up goes up, down goes down, left goes ....

  • Always the same in other dialogs?

Open message boxes and check

  • Is the box big enough that the user can read all of the information without increasing the frame? (Not possible with long texts)

  • Is the text cut by the frame of the box?

Understandable texts

  • Is the caption of the box and the text in the message box understandable to a normal user? (Be aware - not all of our users are experts!)

  • Are the captions of the buttons and the anticipated action after click understandable in any language?

Structure of the text information

  • Is the text well structured by information content

Check the aesthetic aspect of the layout

  • Are the dialog elements pressed together even if there is enough space above and under a group of elements?

  • Are the dialog elements 'glued' to the frame of the dialog area?

Check the grouping aspects of the layout

  • Do you recognize all dialog elements belonging together?

Check the function of 'Back' at the first dialog page and 'Abort'

  • 'Back' cannot jump back but will close the dialog.

  • If user made no changes 'Abort' can close module without warning request. If there are changes a warning pop up is required.

Check the function of 'Back' at the other dialog pages and 'Abort'

  • 'Back' will go back.

  • If user made no changes 'Abort' can close module without warning request. If there are changes a warning pop up is required.


[Abort]-button in all of the configuration modules?

Correct feedback if user wants to 'abort' (some discussion necessary)

Warning pop-up if something will be deleted

No Buttons with status 'deactivated'

Progress-bar to show 'system is working'

Standardized order of buttons (soon this topic will be added to the style guide)

Fine behavior if installation fails

Comfortable size of message boxes with long text

Standardized message texts and button labels and better format of help text (work for 7.3)
