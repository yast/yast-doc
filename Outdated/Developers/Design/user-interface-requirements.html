YaST2: Design: User interface

Purpose

The user interface is used to exchange information with the user. It is used to

- inform the user about 'whats going on'
- presenting data to the user
- getting data from the user
- offering context sensitive help
- asking 'what to do next'

The user interface should be completely independant (and unaware of) the workflow environment. It is controlled by a specific (network capable) command protocol which makes no assumptions about the kind of user-side interface (e.g. text, graphics) available. However, it might give some 'hints' on how to present data (e.g. tabular, horizontal menues, ...) but neither the application (i.e. workflow module) nor the user interface itselft might rely on this.

The separation of form (user interface) and function (workflow) is essential.

Remark: With this concept, a purely optical re-design can be presented as a ´new and enhanced´ version ... for our marketroids.

Design parameters

    text and graphic
      generic syntax to describe 'masks'
      'hints' for presentation are possible
      usable on a teletype (e.g. line-oriented braille converter for visually impaired people)
    Multilingual
      Unicode ?
      gnu gettext ?
    networking
      interface protocol
      transmission of what to present and how to present
    expandable
      transmission of graphics and sounds
      version information in protocol
    texts, strings, numerical values
      input, output
    choices
      checkbox, radiobutton, menue
    presentation
      tables, multiple windows, alert box, progressbar, ...
    help
      at any time (hotkey ! F1 ?), context sensitive
    demo-mode for pictures in the installation manual
      demo-frontend to reproduce every possible mask (filled with reasonable data)
    named masks
      every mask (i.e. view presented to the user) must be non-ambiguously named and reproducable. The names are important for translators as checkmarks in the TeX source of the installation manual
    stand-alone vs. linked masks
      most masks may be used either stand-alone (i.e. user selected just *this* workflow) or 'linked'. Here, linked means being part of a larger workflow with the possibility to go back and forward.
    checklist display
      In a larger workflow, it is desirable to give the user information about the current path through the workflow. For example, a list of 'steps' with green checkmarks for 'step done', yellow mark for 'in progress', etc.
    background graphics
      Help may be provided by background graphics. For example, a zoom-up of the registration page from the manual was suggested by I-Support
    KDE/Gnome
      A special KDE or Gnome version of the user interface is desirable.

interface protocol

The protocol must
  • work over the network
    break data stream in blocks
    master/slave concept
    timeouts, retransmissions
  • be safe
    checksummed
    retransmit on bad checksum
  • be secure
    encoded/crypted (by ssh ?)
  • be small
    compressed data ?
  • two way
    revert master/slave
    interface may send request to module
