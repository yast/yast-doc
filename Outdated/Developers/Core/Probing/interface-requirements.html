YaST2: F-Spec: Autoprobe: Interface

Scope

This document describes a proposal for a high-level interface to the hardware autoprobing.

Requirements

The autoprobe library should scan the hardware once at boot and be prepared to fail. Alternative approaches are to be defined.

The result of the hardware scan is a 'runtime database' which can be queried and changed, i.e. by isapnp -c.
There should be a way to compare two databases to identify changes due to added,changed,removed components (i.e. isdn-card added, graphics card changed, Zip-drive removed).

The autoprope should concentrate on probing hardware to firmware or software. For example, it should identify which devices are connected to a scsi bus but not try to read partition tables from disk drives (this is considered firmware).

Abstraction levels

  • Upper level

    • Highest abstraction, should answer questions like
    - is a mouse attached ?
    - is a network device attached (modem, isdn, ethernet ...) ?
    - do we have local disks ?
    - do we have local cd-rom drives ?

  • Middle level

    • Should answer questions like
    - what kind of mouse ? which device ? (for configuring gpm
    - is a scsi controller attached ?
    - is a modem attached and where ?

  • Lower level

    • Most hardware-dependant level, typical questions are
    - Vendor and type of isdn-card
    - Vendor, type, and capacity of second scsi drive
    - is irq 11 in use ?
